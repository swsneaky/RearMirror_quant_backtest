# 量化回测框架架构研究与 RearMirror 设计建议

## 一、主流量化框架架构对比

### 1.1 框架特性对比表

| 框架 | 架构模式 | 数据模型 | 回测引擎 | 交易成本 | 滑点模拟 | 实盘支持 | 多资产 | 并行化 |
|------|----------|----------|----------|----------|----------|----------|--------|--------|
| **Backtrader** | 事件驱动 | Bar/Tick/OrderBook | 逐Bar迭代 | 可配置佣金模型 | 固定/百分比/自定义 | 完整Broker抽象 | 是 | 单进程 |
| **Zipline** | 事件驱动 | Bar/Tick (Bundle) | 每日/分钟迭代 | 固定佣金 | 简单滑点模型 | 无(仅回测) | 是 | 单进程 |
| **VeighNa** | 事件引擎 | Tick/Bar/Depth | 逐Tick/Bar | 精细化成本 | 支持多档滑点 | 完整Gateway抽象 | 是 | 多进程 |
| **qlib** | 向量化为主 | 表格数据 | 批量矩阵运算 | 固定费率 | 简单模型 | 有实盘接口 | 是 | 多进程/Dask |
| **PyAlgoTrade** | 事件驱动 | Bar/Tick | 逐Bar迭代 | 固定佣金 | 简单滑点 | 有限支持 | 否 | 单进程 |

### 1.2 各框架架构特点

#### Backtrader
```
┌─────────────────────────────────────────────────────────┐
│                    Cerebro (主引擎)                      │
├─────────────────────────────────────────────────────────┤
│  Strategy ◄─── Broker ◄─── Data Feed ◄─── Analyzers    │
│     │            │            │              │          │
│  [notify]    [order]      [Bar]         [metrics]       │
│     │            │            │              │          │
│  next()      execute()    load()        compute()       │
└─────────────────────────────────────────────────────────┘

特点:
- 完整的策略生命周期: prenext -> next -> notify_order
- 支持多数据源同时运行
- 订单状态机完整 (Submitted -> Accepted -> Completed/Cancelled)
- Observer 模式用于实时监控
```

#### Zipline (Quantopian)
```

- 分钟级数据支持
```

### 1.3 架构维度深度对比

#### 数据模型设计
| 模型 | 适用场景 | 内存效率 | 精度 | 代表框架 |
|------|----------|----------|------|----------|
| **Bar (OHLCV)** | 日线/分钟线回测 | 高 | 中 | 所有框架 |
| **Tick** | 高频策略、精确撮合 | 低 | 高 | VeighNa, Backtrader |
| **OrderBook** | 做市策略、Level2 | 极低 | 极高 | VeighNa |

#### 事件驱动 vs 向量化
```
事件驱动:
┌─────────────────────────────────────┐
│ for bar in data:                    │
│     strategy.on_bar(bar)            │
│     broker.execute_pending_orders() │
│     portfolio.update()              │
└─────────────────────────────────────┘
优点: 精确状态管理、易于理解、实盘一致性高
缺点: 慢 (逐Bar循环)、难以并行化

向量化:
┌─────────────────────────────────────┐
│ signals = model.predict(X)          │
│ returns = signals.shift(1) * y      │
│ nav = (1 + returns).cumprod()       │
└─────────────────────────────────────┘
优点: 快 (批量矩阵运算)、易于并行化
缺点: 难以精确模拟订单撮合、滑点模型简单
```

#### 交易成本模型
```
Level 1: 固定费率 (friction_cost = 0.0004)
Level 2: 分类成本 (佣金/印花税/过户费)
Level 3: 冲击成本模型 (√(交易量/日成交额))
Level 4: 完整订单簿模拟
```

### 2.2 核心组件评估

| 组件 | 当前状态 | 优点 | 不足 |
|------|----------|------|------|
| **Pipeline** | 成熟 | 清晰的阶段划分、支持增量计算 | 单向流程、缺乏交互式能力 |
| **Data Layer** | 完善 | 分层设计、版本化资产、SQLite WAL | 无 Tick/OrderBook 支持 |

## 五、总结

### 5.1 RearMirror 定位

**RearMirror 是一个面向中低频多因子策略的向量化回测框架**，具有以下特点:
- 完善的数据分层管理
- 插件化因子注册系统
- Walk-Forward 回测引擎
- 版本化数据资产

### 5.2 与主流框架的差异

| 对比项 | RearMirror | Backtrader/VeighNa |
|--------|------------|---------------------|
| 目标用户 | 量化研究员 | 策略开发者/交易员 |
| 核心场景 | 因子挖掘/组合优化 | 策略开发/实盘交易 |
| 数据粒度 | 日线 Bar | Tick/OrderBook |
| 回测精度 | 向量化近似 | 逐Tick 精确 |
| 实盘能力 | 无 | 完整 |

### 5.3 演进路线

```
Phase 1 (当前): 完善因子研究能力
    └── 优化成本模型、滑点模拟

Phase 2: 增强回测精度
    └── 引入事件驱动核心、订单状态管理

Phase 3: 实盘交易能力
    └── Gateway 抽象、风控模块

Phase 4: 高频数据支持
    └── Tick/OrderBook 数据流、分布式计算
```
