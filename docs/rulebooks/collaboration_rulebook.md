# Collaboration Rulebook (协作明细规则)

本文件承载 RearMirror 三大文件协作的详细操作规则，是 `AI_CONTEXT.md` 的明细分册。

适用范围:
- `HANDOFF.md` 的固定模板、动作类型和最小读取范围写法
- `WORKLOG.md` 的追加格式、记日志时机和质量要求
- 三大文件之间的动作语义、回退语义和校验留痕要求
- prompt 最小化后的角色唤醒约束；真正的角色动作边界以 `docs/roles/` 下对应角色卡片为准

---

## 1. Handoff Payload Standard (高效交接格式)

`HANDOFF.md` 必须使用固定模板，并且只允许记录当前轮次的增量信息，不得重复项目背景、长期规则或历史流水。

### 1.1 Mandatory Rules (强制规则)
- `HANDOFF.md` 必须由发送方整文件覆写，不得在旧内容后追加新块。
- `HANDOFF.md` 第一行必须且只能是状态标签。
- `HANDOFF.md` 必须严格按固定字段顺序书写，不得随意调换字段顺序。
- 若某字段当前无内容，必须填写 `无`，不得省略字段。
- `HANDOFF.md` 只保留“当前球在谁手里、下一步做什么、基于什么证据”的最小必要信息。
- `HANDOFF.md` 不记录历史轨迹；历史轨迹统一写入 `WORKLOG.md`。
- `HANDOFF.md` 必须显式声明当前事项属于哪条轴线、哪个流程或治理切片；不得让接手者自行猜当前工单归属。
- `HANDOFF.md` 必须显式给出 `动作类型:` 字段，其值只能是 `正常流转`、`故障回退` 或 `阻塞待补证据`。
- `HANDOFF.md` 必须给出最小读取范围，用于约束接手者优先读取哪些 `WORKLOG.md` 记录、哪些长期规则切片以及哪些必要代码/测试入口。
- `HANDOFF.md` 必须满足“主宪章后可接手”要求：即使在新 session 中不预读 `WORKLOG.md` 正文，接手者也应能在先读完 `AI_CONTEXT.md` 主宪章后，仅凭 `HANDOFF.md` 判断自己是否该接球、当前属于哪条线/哪个流程、下一步先做什么。
- `HANDOFF.md` 的最小读取范围中，`AI_CONTEXT.md` 不得缺席；至少必须包含一组能覆盖当前球权、阶段归属和日志协议的最小宪法切片。
- `HANDOFF.md` 的最小读取范围中，`WORKLOG.md` 也不得缺席；至少必须指出当前流程、当前治理切片或当前 `round_id` 的相关记录。
- `HANDOFF.md` 的最小读取范围默认应收敛为三类编号项：`WORKLOG.md` 记录、`AI_CONTEXT.md`/rulebook 规则切片、代码/测试/脚本入口；若同类入口超过一个，应尽量在同一编号项内合并表达。
- `HANDOFF.md` 的 `结论`、`必做项` 和 `完成标准` 应尽量同时说明“业务主线停留位置”和“当前工单切片归属”；若当前工单属于治理侧线，不得把治理推进写成业务主线已前进。
- `HANDOFF.md` 的 `必做项` 只允许写当前接收方此轮必须完成的动作；不阻塞本轮阶段验收的残余事项，应迁移到 `docs/open_items.md`，而不是继续挂在当前工单里。
- 若当前事项属于业务主线，且本轮已形成用户可见的定量结果或下一轮方向裁定，则 `HANDOFF.md` 的最小读取范围与证据中必须包含当前轮结果简报路径；不得只给底层 QA 目录而不提供结果摘要入口。
- 后续 prompt 应尽量退化为“角色唤醒语”；角色的允许动作、禁止动作、拿到球后的最小推进义务与非球权待命行为，统一以 `docs/roles/` 下对应角色卡片为准，不再在 prompt 中重复展开。
- 任何角色只要覆写 `HANDOFF.md`，或修改 `AI_CONTEXT.md` / `WORKLOG.md` / 本分册的协作协议相关内容，就必须在正式流转前运行 `tools/validate_three_files.py`；校验未通过时不得交接。

### 1.2 Mandatory Template (强制模板)

```md
[状态标签]

动作类型:
[正常流转 / 故障回退 / 阻塞待补证据]

所属轴线:
[business_mainline / governance_and_operationalization]

当前流程:
[例如 raw_feature / label_and_dataset / cleanup_authorization_and_dryrun]

关联业务主线:
[若当前工单属于治理切片，则写它服务或阻塞的业务主线阶段；无则写 无]

最小读取范围:
1. [接手者必须优先读取的 WORKLOG 范围]
2. [接手者必须优先读取的 AI_CONTEXT / rulebook 章节]
3. [接手者必须优先核对的代码、测试或脚本入口]

结论:
[一句话说明当前是否可继续推进，以及为什么]

必做项:
1. [本轮必须完成的事项]
2. [按优先级继续列出]

证据:
- [失败测试 / 核心报错 / 关键日志 / 相关文件]

完成标准:
- [达到什么条件后才允许交给下一个角色]

非目标:
- [本轮明确不要做的事，防止范围蔓延]

发送方:
[角色]

接收方:
[角色]
```

补充规则:
- `HANDOFF.md` 可以过期；它本来就是当前轮次工单，不是长期知识库。
- `HANDOFF.md` 只有一个模板，但必须明确当前动作类型是正常流转、故障回退还是阻塞待补证据。
- `动作类型:` 必须与 `结论` 第一行前缀保持一致；禁止字段写“正常流转”而结论实际写成“故障回退”。
- `结论` 第一行必须直接写明动作类型，禁止让接收方自行猜测这是继续推进还是先修问题。
- `所属轴线` 只能填写 `business_mainline` 或 `governance_and_operationalization`。
- `当前流程` 必须优先使用 `AI_CONTEXT.md` 中项目阶段模型定义的正式阶段名或治理切片名。
- 若 `所属轴线=governance_and_operationalization`，则 `关联业务主线` 不得省略；若当前治理事项不直接服务或阻塞业务主线，也必须写 `无`。
- `最小读取范围` 应以“够接手、不过量”为原则，避免要求接手者全文阅读 `WORKLOG.md`。
- `最小读取范围` 是补充读取，而不是前置替代；接手顺序始终先从 `AI_CONTEXT.md` 主宪章开始，再读 `HANDOFF.md` 本身，然后才补读其他文档。
- `最小读取范围` 默认至少应包含 `AI_CONTEXT.md` 的 `0.5 Reading Scope Policy` 与 `1.0~1.5 Collaboration Constitution`；若当前事项需要详细协议，再补本分册相应章节；若涉及文件治理，还应补 `docs/rulebooks/engineering_constraints.md` 的 `5. File Lifecycle & Cleanup Governance`。
- 每条结论都应尽量附带测试名、日志片段、文件路径或可复现实验入口。
- 允许简洁，禁止空话；允许结论，禁止无证据判断。
- 允许的角色值仅限：`Session A`、`Session B`、`Session C`、`Session D`。
- `必做项` 应只包含当前接收方必须完成的动作，不得混入未来轮次工作。
- `完成标准` 必须可验证，优先使用测试名、脚本名、数据检查条件或显式放行条件表达。

### 1.3 Temporary User Override Realignment (用户临时改派后的协议回正)
当用户显式要求某个角色临时越过当前 `HANDOFF.md` 球权去执行动作时，允许先执行，但必须在动作结束后做协议回正，避免 4 个对话逐渐脱离正式阶段模型。

强制规则:
- 用户显式改派只覆盖“当前这一次动作”，不自动改写后续正式球权。
- 动作完成后，执行者必须先在 `WORKLOG.md` 追加一条记录，明确写出“用户直接改派/显式覆盖”、实际执行了什么、为什么这样执行。
- 随后必须把 `HANDOFF.md` 回正到当前真实状态：谁该接球、业务主线停在哪一阶段、当前工单属于哪个阶段或治理切片。
- 若用户改派后的结果改变了阶段结论，`HANDOFF.md` 与 `WORKLOG.md` 必须在同一轮内完成同步；不得长期依赖聊天上下文维持真实状态。
- 若用户改派只是临时补证，不改变当前阶段结论，则应在 `WORKLOG.md` 留痕后恢复到原先应有的接收方，而不是默认把球权永久转给临时执行者。

### 1.4 Action Types (动作类型)
`HANDOFF.md` 的动作类型必须在语义上明确区分，但不需要拆成多份文档或多套模板。

1. 正常流转
   - 含义：当前阶段已完成或已达到本轮可接受状态，球正常传给下游角色继续推进。
   - 写法：`结论` 应明确写出“当前阶段已完成/已通过，下一棒是谁，为什么交给他”。
   - `必做项`：只写下游角色当前必须完成的动作，不得混入上游修复项。

2. 故障回退
   - 含义：当前阶段执行失败、证据表明实现或配置有问题，球必须退回上游角色修复。
   - 写法：`结论` 应明确写出“当前阶段失败或不满足继续条件，必须退回给谁，失败点是什么”。
   - `必做项`：聚焦修复项、补证项和重新放行前必须完成的验证，不得假装继续推进下游阶段。

3. 阻塞待补证据
   - 含义：当前既不能判定通过，也不能直接定位为实现缺陷，需先补充证据、阶段定义或运行事实。
   - 写法：`结论` 应明确写出“当前阻塞，缺什么证据/定义，补齐前不得推进”。
   - `必做项`：只写补证动作，不得提前写下游开发或执行动作。

约束:
- 正常流转和故障回退必须区分语义，但都使用同一份 `HANDOFF.md` 模板。
- `接收方` 由动作类型决定：正常流转通常交下游角色，故障回退通常交上游角色，阻塞待补证据交最适合补齐证据或重新裁决的角色。
- 若 `HANDOFF.md` 未明确动作类型，应视为不合格交接单。

### 1.5 Residual Issue Register (残余事项登记)
当某个问题已不再阻塞当前阶段继续推进、打回或收口时，不应继续把它滞留在当前 `HANDOFF.md` 里反复传球，而应转记到 `docs/open_items.md`。

强制规则:
- 只有“当前仍阻塞阶段结论”的问题，才能留在 `HANDOFF.md` 的 `必做项` 中。
- 已降级为后续治理、兼容性收尾、归档清理或体验优化的问题，应迁移到 `docs/open_items.md`。
- 迁移时，`WORKLOG.md` 应记录迁移依据：为什么它不再阻塞当前阶段、后续应由谁先接。
- `docs/open_items.md` 不是 backlog 大全；只收录和当前阶段推进直接相关、且容易在多角色流转中丢失归属的残余问题。

## 2. Work Log Protocol (工作日志协议)

`WORKLOG.md` 是追加式审计轨迹，不允许按轮次整文件覆写。它用于回答以下问题：
- 谁在什么时候接手了哪一阶段
- 实际执行了什么动作
- 基于什么证据做出通过、打回或阻塞判断
- 本轮迭代是如何推进或回退的

### 2.1 Core Rules (核心规则)
- `WORKLOG.md` 必须采用追加式写入；新记录必须追加到文件尾部，不得插入到历史记录中间。
- `WORKLOG.md` 记录事实轨迹，不替代 `HANDOFF.md` 的当前状态职责。
- `HANDOFF.md` 可以过期；`WORKLOG.md` 不应因状态切换而丢失历史信息。
- `WORKLOG.md` 重点记录动作、证据和结论，不记录闲聊、情绪化评论或无证据判断。
- `WORKLOG.md` 中每条记录必须完整自洽，允许脱离上下文单独阅读。

### 2.2 Mandatory Entry Format (强制记录格式)
每条工作日志记录必须严格按以下模板追加：

```md
## [时间戳] | [角色] | [阶段] | [状态]
- round_id: [本轮唯一标识]
- action: [本次执行的动作]
- inputs: [本次基于哪些文件、配置、测试、脚本或数据]
- findings: [观察到的事实或发现的问题]
- evidence: [测试名、日志片段、文件路径、关键输出]
- decision: [通过 / 打回 / 阻塞 / 继续观察]
- next: [下一步交给谁或下一动作是什么]
```

字段说明:
- `时间戳`：精确到分钟，便于审计顺序。
- `角色`：Session A / B / C / D 中之一。
- `阶段`：优先使用 `AI_CONTEXT.md` 中项目阶段模型定义的正式阶段名或治理切片名。
- `状态`：例如 `open`、`in_progress`、`blocked`、`passed`、`passed-with-conditions`。
- `round_id`：同一轮迭代内跨角色共享的标识，便于追踪多次打回。
- 若某字段当前无内容，必须填写 `无`，不得删字段。
- `evidence` 至少包含以下之一：测试名、脚本名、日志片段、文件路径、命令输出摘要。
- `decision` 只能写明确判定，不得写模糊措辞，如“可能可以”“先看看再说”。

### 2.3 Logging Triggers (强制记日志时机)
以下动作发生时，必须追加一条 `WORKLOG.md` 记录：
- 角色切换或正式交接前后
- 阶段被打回、阻塞、放行时
- 运行了关键测试、关键审计检查、关键数据验收脚本后
- 修改了长期规则、阶段门禁、数据层契约、工程治理边界或实验边界后
- 发现会影响本轮推进结论的新证据后

默认时机:
- `WORKLOG.md` 默认在完成一个可判定动作之后填写，而不是在接手前预写。
- 至少应在完成执行、验证、审计或规则修订并形成结论后，再把动作、证据和决定追加到 `WORKLOG.md`。

以下动作可不单独记日志：
- 纯格式整理、无结论价值的微小文案修改
- 未形成结论的临时探索性阅读

### 2.4 Relationship With Handoff (与交接单的关系)
- `HANDOFF.md` 只回答“当前球在谁手里、下一步做什么”。
- `WORKLOG.md` 只回答“这球之前怎么传过来、为什么这样传”。
- 默认读取顺序是：先 `AI_CONTEXT.md`，再 `HANDOFF.md`，最后才按需进入 `WORKLOG.md`。
- 任何会影响阶段结论的 `HANDOFF.md` 更新，都应在 `WORKLOG.md` 留下一条对应记录。
- 若本轮覆写 `HANDOFF.md` 或修订了三大文件协议，`WORKLOG.md` 对应记录应写明 `tools/validate_three_files.py` 是否通过，以及它拦下或确认了什么。

### 2.5 File Header (文件头格式)
`WORKLOG.md` 文件头必须固定为：

```md
# WORKLOG

[后续所有日志记录按时间顺序追加到文件尾部]
```

### 2.6 Quality Requirements (质量要求)
- 每条日志必须让第三方在脱离上下文的情况下理解本次动作、证据与结论。
- 允许简洁，但不得省略证据来源。
- 若记录结论为“打回”或“通过”，必须明确写出判定依据。

### 2.7 Targeted Reading Rule (定向读取规则)
- `WORKLOG.md` 默认不做全文读取；接手者应先按 `HANDOFF.md` 的 `所属轴线`、`当前流程`、`关联业务主线` 和 `最小读取范围` 定向读取相关记录。
- 若当前事项属于业务主线，优先读取：同一 `当前流程` 的最新连续记录、同一 `round_id` 的最近往返记录、以及最近一条阶段启用或阶段收口记录。
- 若当前事项属于治理侧线，优先读取：同一治理切片的最新连续记录、最近一条定义该切片边界的 A 记录，以及被该治理切片服务或阻塞的业务主线桥接记录。
- 只有当定向读取后仍无法判断球权、阶段归属、回退条件或证据链时，才向前扩大读取范围；若仍无法判断，再升级为全文复读或回到 Session A / Session C 补定义。
- `HANDOFF.md` 的发送方负责把最小读取范围写到足以让接手者工作；接收方不应在未读完该最小范围时宣称已完成接手。

### 2.8 Stage Gate Recording (阶段门禁记录)
- 当某一业务阶段准备交给下一角色时，`WORKLOG.md` 的 `findings` 或 `decision` 应尽量对照该阶段在 `docs/rulebooks/project_baseline.md` 中的门禁卡，明确写出“进入前提是否满足、交付物是否具备、最低证据是否齐全、是否允许前进”。
- 若当前只完成了“能力已具备”或“本轮验证已通过”，不得在日志中写成“阶段已收口”；只有 A 的阶段收口记录可以写“阶段结束/进入下一阶段”。

### 2.9 Result Output Recording (结果输出留痕)
- 任何业务主线轮次只要已经形成用户可见的定量结果，Session D 就必须在当前 QA 目录中留下结果简报，并在 `WORKLOG.md` 的 `evidence` 中显式引用它。
- Session C 审计业务主线轮次时，不仅要审底层 JSON / Parquet / 日志，也要审结果简报是否与底层证据一致；若结果简报缺失或夸大结论，应视为审计缺口。
- Session A 在做“阶段收口 / 下一阶段激活 / 下一轮因子调整”裁定时，必须引用当前轮结果简报；不得只凭 `analysis_summary.json` 或 `metrics_summary.json` 的文件存在性直接推进方向决策。
- 结果简报的详细格式与分阶段最低要求，统一以 `docs/rulebooks/result_reporting.md` 为准。

### 2.10 Strategic Branching After Decision-Ready Rounds (可决策轮次后的战略分流)
- 当某一业务主线轮次已经满足结果简报门禁，且 A 正准备据此做“继续主线 / 下一轮因子调整 / 阶段收口”裁定时，A 还必须额外回答一次能力演进问题：本轮之后是否需要激活一个 roadmap capability slice。
- 允许的 roadmap capability slice 至少包括：
  - `iteration_artifacts_and_result_routing`
  - `multimodel_hyperparameter_optimization`
  - `model_stacking`
  - `frontend_workbench`
- A 的裁定只能是以下两类之一：
  - `defer_capability_slice`：当前仍以业务主线推进为主，相关能力项继续留在 `docs/open_items.md`
  - `activate_capability_slice`：显式开启一个能力切片，并通过新的 `HANDOFF.md` 把球交给对应角色
- 若 A 选择 `defer_capability_slice`，`WORKLOG.md` 必须写清楚延后原因，避免团队长期默认“永远只迭代主线”。
- 若 A 选择 `activate_capability_slice`，`HANDOFF.md` 必须显式写出：
  - 当前能力切片名称
  - 它服务的业务主线阶段
  - 为什么现在激活，而不是继续纯主线迭代
- Session B / C / D 不得自行从当前业务迭代跳出并启动新能力建设；但当 A 已完成上述显式激活后，后续角色也不应继续假设“这些目标只是聊天意图”。
- capability slice 的默认顺序、进入前提与长期裁定口径，统一以 `docs/rulebooks/capability_rollout.md` 为准。

---

## 3. Rollback Rules (打回规则)

- 打回是 `HANDOFF.md` 的一种故障回退动作类型，不是独立文档流程。
- 测试失败：Session D 将状态改回 `[WAITING_FOR_B_CODER]`，并附失败测试、核心断言与最短可复现线索。
- 审计失败：Session C 将状态改回 `[WAITING_FOR_B_CODER]`，并附驳回依据、违反的长期规则、重新放行条件。
