# Engineering Constraints (工程约束)

本文件承载 RearMirror 的长期工程约束、运行约束和文件治理规则，是 `AI_CONTEXT.md` 的工程治理分册。

---

## 1. Incremental Computing (增量计算)
1. 增量计算的目标是减少重复工作，而不是改变语义。
2. 增量路径必须明确 `warmup` 边界，避免窗口算子在切片边界失真。
3. 若增量路径与全量路径需要不同实现，必须优先验证两者输出一致性。

## 2. Performance Guidelines (性能原则)
1. 避免在滚动窗口主循环中调用 `gc.collect()` 这类高阻塞操作。
2. 优先使用向量化算子、累计量公式、批量窗口运算，谨慎使用 Python 层 `rolling.apply`。
3. 任何窗口无关的中间变量都应尽量外提并复用，减少重复计算。
4. 若采用近似算法替代精确算法，必须满足以下至少两项：
   - 对树模型或当前下游任务影响可接受
   - 有基准测试或误差评估支撑
   - 有配置或文档可追溯

## 3. Runtime Modes & Degradation (运行模式与降级策略)
1. 正式全量链路若包含大规模 versioned asset 写入、compat 别名刷新、compat parquet 导出等多个高成本动作，不应默认视为单一不可拆分步骤。
2. 对资源敏感的阶段，系统设计应允许把“核心正式资产生成”和“兼容性收尾/导出”拆成不同运行模式或不同批次执行。
3. 日常研究使用应优先保证可持续运行和正式资产闭环，不应为了兼容副产物把共享环境拖入内存或 I/O 崩溃区间。
4. 若 compat 产物仅服务旧接口、过渡期脚本或展示层，则其优先级不得默认高于正式 versioned asset 闭环。
5. 一旦某运行模式在共享机器上稳定触发资源退化，必须显式标记为“非默认模式”，并给出替代入口或执行前提。
6. 任何降级策略都必须保持语义可追溯，禁止用“少写一点、先跳过去”这类不可审计方式绕过正式资产一致性。

## 4. Compatibility & Logging (兼容性与日志)
1. 面向 Windows 的输出必须考虑控制台编码差异，日志文本应优先保证稳定打印。
2. 日志可以帮助定位，但不得成为流程中断源。
3. 人类可读性优先于花哨格式；日志应以排障效率为第一目标。

## 5. File Lifecycle & Cleanup Governance (文件生命周期与清理治理)
RearMirror 必须对开发过程文件、运行产物和临时输出实行显式分类、显式生命周期和显式删除授权，禁止继续依赖“谁顺手生成、谁看着处理”的隐式习惯。

### 5.1 File Classes (文件类别)
项目文件至少分为以下七类：
1. 正式资产：正式数据库、正式 versioned asset、正式 compat parquet、正式模型与正式分析产物。
2. 实验产物：按实验或任务隔离的配置快照、结果、指标、持仓、报表和实验日志。
3. QA 临时产物：smoke run、中样本验证、沙盒数据库、沙盒 parquet、一次性核对输出。
4. 缓存：可由上游数据或正式资产重建的缓存文件、临时中间结果和下载缓存。
5. 日志：正式运行日志、QA 日志、任务日志、排障日志。
6. 工具与草稿脚本：一次性排障脚本、临时验证脚本、开发辅助工具。
7. 大文件：体积显著的大型 parquet、数据库、模型或导出文件；大文件本身不是独立语义类别，必须同时归属于上述某一正式类别。

### 5.2 Lifecycle Rules (生命周期规则)
1. 正式资产：默认保留，禁止自动删除；只允许显式归档、显式替换或在专门治理阶段中按规则迁移。
2. 实验产物：默认按实验目录隔离保留，禁止默认覆盖历史实验；允许归档，不允许无留痕批量删除。
3. QA 临时产物：必须与正式资产隔离；默认可清理，但必须先完成证据留痕；不得写入正式 DB、正式 parquet、正式模型目录。
4. 缓存：必须标记为可重建对象；允许清理，但前提是其上游底稿或正式资产仍在，且清理不会破坏当前活动阶段。
5. 日志：必须分层存放；允许轮转、归档和按规则清理；禁止删除当前活动运行所依赖的日志。
6. 工具与草稿脚本：不得继续在仓库根目录无限扩散；应收敛到专用目录，且不得被误认成正式入口。
7. 大文件：若属于正式资产或实验产物，则继承对应保留规则；若属于 QA、缓存或过期日志，则可纳入安全清理范围，但必须先确认没有被当前阶段引用。

### 5.3 Path Discipline (路径纪律)
1. 新增输出必须落入明确分层路径，不得直接把临时文件、QA 产物或一次性脚本输出写到仓库根目录。
2. 正式资产层、实验产物层、QA 临时层、日志层、缓存层、工具/草稿脚本层必须在路径上可区分，禁止混写。
3. QA 路径必须默认与正式路径隔离；任何临时验证脚本若会产生产物，必须优先写入 QA 层而不是复用正式目录。
4. 根目录仅允许长期正式入口和长期说明文件；一次性排障脚本、临时运行脚本和阶段性辅助脚本应收敛到专用目录。

### 5.4 Deletion Authority (删除授权)
1. A 定义删除边界：A 负责定义哪些类别允许清理、哪些类别只允许归档、哪些类别绝对禁止自动删除。
2. B 实现删除机制：凡涉及清理脚本、归档脚本、路径重构、旧兼容路径退役、白名单/黑名单、安全检查或 dry-run 机制，均属于 B 的实现职责。
3. D 执行批准内清理：对已批准的 QA 临时产物、缓存、过期日志和其他可重建对象，D 可以按既有机制执行清理，并负责记录证据。
4. C 审计删除边界：C 不负责执行删除，但负责审查删除规则是否越权、是否误伤正式资产或实验证据。
5. 开发型删除归 B：凡删除动作属于重构、替换、删旧代码、删兼容逻辑或迁移的一部分，默认归 B，不视为 D 的日常清理权限。

### 5.5 Cleanup Triggers (清理触发时机)
1. QA 阶段完成后，必须决定本轮 QA 产物是保留到专用 QA 目录还是按规则清理；禁止长期堆放在模糊路径下。
2. 新机制落地后，旧的临时输出路径必须有退役动作，至少要明确“迁移、归档或删除”三选一，不得无限并存。
3. 当某类缓存、日志或 QA 大文件已不再服务当前阶段且已有必要证据留痕时，应在当前轮次或下一轮交接中明确是否清理，而不是无限延期。
4. 任何批量清理、批量迁移或批量归档动作都必须先具备 dry-run 或等价预演能力，且执行结果必须写入 `WORKLOG.md`。

### 5.6 Evidence Requirements (留痕要求)
1. 发生清理、归档、迁移或路径退役时，`WORKLOG.md` 必须记录对象类别、作用路径、执行方式、是否 dry-run、是否真正删除以及影响范围。
2. `HANDOFF.md` 若把清理任务交给下一角色，必须明确写出动作类型、允许清理的类别和禁止触碰的路径边界。
3. 未经 `HANDOFF.md` 或既有长期规则授权，不得对正式资产、实验产物或阶段证据文件执行删除。

### 5.7 Three Core Files (三大文件自身约束)
1. `AI_CONTEXT.md` 属于长期规则文件，禁止作为“清理对象”做日常删除或轮转；仅允许按长期维护规则修订。
2. `HANDOFF.md` 是单文件当前工单，允许整文件覆写，但不属于“删除后重建”的普通临时文件。
3. `WORKLOG.md` 是追加式审计轨迹，禁止用“清理日志”名义删除历史记录；若未来需要归档，只能在专门治理阶段中采用“保留索引 + 分卷归档”的方式处理。
