# Project Baseline (项目基线)

本文件承载 RearMirror 相对稳定的项目基线、数据层级和端到端工作流定义，是 `AI_CONTEXT.md` 的基线分册。

---

## 1. Project Goal (项目目标)
RearMirror 是一个面向中国 A 股市场的多因子量化研究与回测平台。项目目标是通过 YAML 配置驱动，串联数据下载、规范化校验入库、特征工程、截面处理、模型训练、Walk-Forward 回测、因子分析、实验管理与结果可视化，保证实验可追踪、结果可复现、流程可增量运行。

## 2. Tech Stack & Conventions (技术栈与约定)
- Python 3.10+
- 主要依赖：`pandas`, `numpy`, `pyarrow`, `pyyaml`, `scipy`, `lightgbm`, `xgboost`, `scikit-learn`, `baostock`, `akshare`, `tqdm`, `plotly`, `joblib`, `shap`, `fastapi`, `uvicorn`
- 数据层设计目标：`SQLite` 为主存储，`Parquet` 作为兼容导入导出与兜底载体
- 配置来源：所有关键参数统一通过 YAML 管理，禁止在业务流程中散落硬编码
- 命名约定：
   - 特征列：`feat_` 前缀
   - 标签列：`label_` 前缀
   - 原始数据列：`raw_` 前缀

## 3. Core Modules (核心模块)
- `pipeline.py`：主流程编排入口
- `src/config_loader.py`：配置加载与合并，参数唯一入口
- `src/data_layer/`：资产 ID、规范化数据层、特征/标签/数据集存储、SQLite 管理
- `src/feature_engine.py`：特征工程工厂、因子注册、增量计算编排
- `src/factors/`：因子实现与扩展入口
- `src/cross_section.py`：截面去极值、行业中性化、标准化
- `src/ml_core/backtest.py`：Walk-Forward 回测引擎与交易约束模拟
- `run_experiment.py`, `task_cli.py`, `src/experiment_store.py`：实验调度、任务执行、结果落盘与血缘追踪

## 4. Data Layer Hierarchy (数据层级定义)
RearMirror 的数据层级必须保持清晰分层，各层职责如下：

1. Raw Layer (原始层)
   - 主载体：按股票分片的原始缓存文件，例如 `stock_daily_cache/<code>.parquet`
   - 辅助载体：下载元数据、缓存索引、数据源返回痕迹
   - 角色：保存下载得到的原始增量数据，作为可追溯、可重放、可审计的底稿
   - 特征：允许保留上游字段口径与下载痕迹，不要求直接适配研究流程
   - 约束：原始层必须长期保留；下游研究流程不得直接将 raw 文件作为正式输入

2. Canonical Layer (规范化层)
   - 主载体：`SQLite` 中的 `daily_bar`、`industry_map`、`index_bar` 等标准表
   - 辅助载体：用于 fallback、导入导出或重建的 `Parquet` 中间产物
   - 角色：作为系统内唯一可信的标准市场数据层，向特征、标签、回测提供统一 schema
   - 必备属性：主键稳定、字段命名统一、类型稳定、量纲固定、可增量 upsert、可追溯来源
   - 准入条件：只有通过规范化、校验、去重与主键处理的数据，才允许进入 canonical 层
   - 读取规则：特征、标签、数据集构建、回测分析默认只读取 canonical 层，不直接读取 raw 文件

3. Feature Layer (特征层)
   - 主载体：版本化特征表、特征集资产标识、特征集组成关系
   - 辅助载体：原始特征矩阵导出文件、调试用中间文件
   - 角色：在 canonical 层之上派生研究特征，不负责修复底层原始数据问题
   - 子层说明：
     - 原始特征矩阵：全量、未中性化、可复用的特征缓存，属于 feature layer 的 staging/cache 子层，不属于 canonical 层
     - 版本化特征表：从原始特征矩阵中按因子选择、处理规则、配置快照生成的正式特征资产
   - 版本化目的：允许不同因子组合、处理中参数、研究方案并存，保证实验复现，避免新实验覆盖旧实验的正式特征集
   - 约束：特征层不得自行发明底层行情口径；任何行情、行业、交易状态解释都以 canonical 层为准

4. Label Layer (标签层)
   - 主载体：版本化标签表、标签集资产标识、标签配置快照
   - 辅助载体：标签导出文件、分析中间产物
   - 角色：在明确预测目标与时间对齐规则后生成标签，不直接绕过 canonical 层读原始缓存
   - 作用：为每个 `date + code` 样本定义监督学习目标，回答“这一行特征要预测什么未来结果”
   - 约束：标签语义、horizon、gap 与训练窗口必须可追溯

5. Dataset & Experiment Layer (数据集与实验层)
   - 主载体：训练集拼装结果、预测结果、持仓、净值、评估指标、实验血缘记录
   - 辅助载体：实验导出文件、可视化缓存、报表副本
   - 角色：组合 feature layer 与 label layer，支撑建模、回测、评估与复现
   - 约束：该层只消费上游标准资产，不承担数据修复职责

## 5. Feature/Label Pairing Contract (特征-标签配对契约)
Feature layer 与 Label layer 的关系必须满足以下约束：

1. 独立版本线：版本化特征表与版本化标签表是两条独立版本线，彼此平行存在，不互相包含，也不互相覆盖。
2. 统一主键：二者必须以 `date + code` 作为对齐键，由数据集构建阶段完成 join。
3. 统一基础 universe 定义：准备配对的 feature_set 与 label_set 必须共享同一套基础股票池定义、时间口径和 canonical 数据来源。
4. 允许边界差异：由于特征 warmup、标签 horizon、停牌缺失、样本过滤等原因，特征表与标签表不要求逐行完全相等。
5. 差异必须可解释：任意 `date + code` 键差异都必须能够由 warmup、horizon、样本过滤、停牌或配置规则解释，不能出现无来源的 universe 漂移。
6. 配对前校验：进入训练集或分析集构建前，必须检查 feature_set 与 label_set 的基础 universe 是否一致，并输出交集、左独有、右独有样本统计；若差异超出预期，应视为数据问题而非默认容忍。

## 6. Canonical Contract (Canonical 层契约)
Canonical 层必须满足以下契约：

1. 唯一入口：对下游研究流程而言，canonical 层是唯一正式市场数据入口。
2. 单一事实表语义：`daily_bar` 等表定义的是“已标准化、已校验、可研究”的市场事实，而不是下载缓存副本。
3. 增量主路径：日常路径应为“下载增量 -> 规范化校验 -> 按主键增量 upsert 到 canonical”，而不是每次都依赖全量合并脚本重建。
4. 原始层保真：raw layer 长期保留，用于追溯、重放、审计；canonical 层不替代 raw layer。
5. 重建工具降级：全量 merge/重建脚本只用于历史修复、口径迁移、灾难恢复、审计复跑，不定义日常增量主路径。
6. 失败即阻断：canonical 入库失败、主键异常、关键字段异常时，应视为数据层失败，不得默默降级后继续主流程。

## 7. End-to-End Workflow (端到端回测主流程)
RearMirror 的标准研究与回测流程按以下顺序推进：

1. 数据下载与缓存更新：从外部数据源拉取原始行情、行业映射、指数数据，并更新本地缓存。
2. 规范化校验入库：将原始下载结果执行字段映射、类型统一、量纲修正、主键检查、去重与增量 upsert，写入 canonical 层。
3. 原始数据增量更新验证：先验证已下载原始数据的增量更新结果、字段语义、格式规范、存储方式是否符合要求；未通过时不得进入后续阶段。
4. 原始特征提取：基于 canonical 原始数据计算未中性化的原始特征矩阵，并保证增量路径与全量路径语义一致。
5. 因子筛选与研究范围定义：确定本轮使用的因子集合、研究区间、回测区间及相关实验配置。
6. 截面中性化：仅对已选因子、在已定义的研究或回测时间区间内执行去极值、行业中性化和标准化。
7. 标签生成与数据集构建：生成标签矩阵，并按特征集、标签集、时间窗口拼装训练集与分析集。
8. 模型训练与 Walk-Forward 回测：按滚动窗口训练模型、生成预测、模拟交易约束与成本，输出持仓、净值和指标。
9. 结果评估与因子分析：评估收益、回撤、夏普、分层表现、IC/ICIR 等指标，并据此决定是否进入下一轮因子调整。
10. 实验落盘与可视化：对配置快照、资产标识、实验结果、分析结果进行持久化，供任务系统和 Dashboard 使用。

## 8. Workflow Gates (流程门禁)
1. 全流程第一道门禁是原始数据增量更新与规范性验证，而不是特征开发、模型训练或结果评估。
2. 只有当原始数据层通过增量一致性检查，且现有数据格式、字段语义、存储方式均符合规范后，才允许进入特征提取验证或开发。
3. 中性化的前置条件是：已完成因子选择、已确定研究或回测时间区间、已明确实验配置；在此之前不得启动中性化。
4. 下游阶段不得绕过上游验收；若原始数据层或特征层未通过验收，则中性化、建模、回测、结果评估一律视为前提不足。
5. 每进入一个新阶段，必须先由 A 明确写出该阶段的目标、完成标准、非目标与回退条件；未经 A 启动，不得默认进入下一阶段。
6. 在单个阶段内部，B 负责开发或修复能力，D 负责调用系统执行该阶段并完成测试、产物落盘和证据记录，C 负责审计复核，A 负责最终收口。
7. C 完成终审复核后，不得直接把“下一阶段开发”或“下一阶段执行”交给 B 或 D；正确顺序是先回到 A，由 A 定义下一阶段，再分配给 B 或 D。
8. 若某阶段系统能力已经完备、无需开发，A 可以直接把该阶段交给 D 执行；但此例外必须由 A 明确写入 `HANDOFF.md`，不得由 B、C、D 自行推定。

## 9. Business Stage Gate Cards (业务阶段门禁卡)
以下门禁卡用于帮助 4 个对话按项目阶段模型推进，而不是在每轮聊天中临时发明“这个阶段算不算完成”。它们是人工执行门禁，不要求一键自动化，但要求在 `HANDOFF.md` / `WORKLOG.md` 中可明确对照。

### 9.1 `raw_to_canonical`
- 进入前提:
  - 已明确数据源、时间范围、基础股票池和 canonical 目标 schema
  - A 已定义本阶段是否以增量 upsert 还是重建工具为主
- 本阶段交付物:
  - canonical 表已可供下游读取
  - raw layer 与 canonical layer 的边界清楚
- 最低证据:
  - 主键/去重检查
  - 关键字段语义、类型和时间范围检查
  - 入库失败时的阻断证据
- 允许前进条件:
  - `daily_bar` / `industry_map` / 相关 canonical 表已满足下游最小输入契约
  - 无未解释的 universe 漂移或关键字段异常

### 9.2 `raw_feature`
- 进入前提:
  - `raw_to_canonical` 已通过门禁
  - 已明确本轮 active factors、窗口和缓存路径策略
- 本阶段交付物:
  - 原始特征矩阵可稳定生成
  - 增量路径与全量路径语义边界清楚
- 最低证据:
  - 原始矩阵列集、日期范围、股票范围检查
  - 增量/全量一致性或可解释差异证据
  - 缓存命中与失效条件说明
- 允许前进条件:
  - 原始特征矩阵已可作为后续中性化输入
  - 当前差异若存在，必须可由 warmup、缓存失效或因子配置解释

### 9.3 `factor_selection_and_neutralize`
- 进入前提:
  - 原始特征矩阵已具备
  - 已明确选用因子组、排除项和截面处理参数
- 本阶段交付物:
  - 版本化 feature_set 与 label_set 或等价正式资产
  - 当前选用因子的中性化矩阵
- 最低证据:
  - 因子筛选结果
  - 中性化后分布/主键/列集检查
  - 正式资产与 compat 产物边界说明
- 允许前进条件:
  - 正式 versioned asset 闭环已成立
  - 若 compat 别名或 compat parquet 仍未完全同步，必须明确写成残余治理事项，而不是假装全链路已完全收口

### 9.4 `label_and_dataset`
- 进入前提:
  - feature_set 已确定
  - 标签定义、horizon、gap 与基础 universe 已明确
- 本阶段交付物:
  - 标签资产
  - 可供训练/分析的 dataset join 结果
- 最低证据:
  - feature/label 主键对齐统计
  - 交集、左独有、右独有样本解释
  - 数据集列集与样本范围检查
- 允许前进条件:
  - feature/label 差异可解释
  - 训练集与分析集构建结果已满足建模入口要求

### 9.5 `train_and_backtest`
- 进入前提:
  - dataset 入口已通过
  - 模型、训练窗口、gap、step、top-k 和摩擦成本口径已冻结
- 本阶段交付物:
  - 预测、持仓、净值、指标与实验血缘
  - 必要时包含模型制品与训练元信息
- 最低证据:
  - 回测运行记录
  - 指标摘要
  - 实验血缘或配置快照
- 允许前进条件:
  - 结果可复现到明确配置与资产版本
  - 没有因数据集错配或运行模式错判导致的伪结论

### 9.6 `analysis_and_delivery`
- 进入前提:
  - 回测结果已具备
  - 因子分析范围、交付对象和输出口径已明确
- 本阶段交付物:
  - IC/ICIR/Decay/SHAP 等分析结果
  - 面向人类决策的结论或交付说明
- 最低证据:
  - 关键指标表或图
  - 结论对应的数据来源说明
  - 是否建议进入下一轮因子调整的判断依据
- 允许前进条件:
  - 输出已能支撑“继续、回退还是重设计”的决策
  - 若只完成了分析产物生成，但未形成阶段结论，不得写成主线已自动前进

## 10. Governance Triage Rule (治理事项分流规则)
治理事项不是越多越好，而是必须服务主线推进。对 4 个协作对话而言，以下分流规则用于避免治理工单无限抢占业务主线。

1. 若治理问题直接阻塞当前业务阶段门禁，则允许进入当前 `HANDOFF.md` 作为当前工单的一部分。
2. 若治理问题只影响后续维护性、兼容性、清理整洁度或下一轮体验，但不阻塞当前阶段结论，则应迁移到 `docs/open_items.md`。
3. 若治理问题已经升级到运行模式、资源预算、正式口径或阶段拆分方式，应触发回到 Session A，而不是继续让 B / D / C 局部修补。
4. 任何治理事项一旦被写成“当前阻塞”，都应明确写出它阻塞的是哪一个业务阶段门禁，而不是只写一个治理切片名。

## 11. Iteration Result Output Gate (迭代结果输出门禁)
业务主线不是只为角色流转服务，还必须为决策者输出结果。每一次业务迭代在进入“阶段收口 / 下一阶段激活 / 下一轮因子调整”前，都必须留下可直接阅读的结果简报。

1. 若当前轮只有底层 QA 产物、原始 Parquet、SQLite 留痕和日志，而没有面向人的结果简报，则默认不能把该轮视为“已完整汇报结果”。
2. `factor_selection_and_neutralize` 的结果输出，必须说明本轮删了什么、保留了什么、形成了哪组正式资产，以及与上一轮相比到底改了什么。
3. `label_and_dataset` 的结果输出，必须说明配对统计、数据集形状、样本范围与 runtime 边界。
4. `train_and_backtest` 的结果输出，必须说明核心回测指标、运行耗时、关键产物是否完整；若是多模型轮次，必须给出对比而不是只报单模型。
5. `analysis_and_delivery` 的结果输出，必须说明当前结论、推荐动作与推荐依据；若 A 准备据此进入下一轮因子调整，则必须能从结果输出中直接读出删因子理由。
6. 结果输出的详细格式、最小文件名和角色责任，以 `docs/rulebooks/result_reporting.md` 为准。
